Architecture Story

Workflow-Driven Architecture

Từ một prompt chung, AgentSwarm biến execution thành chuỗi bước rõ ràng: có planning, có routing, có workflow chuyên trách, có reviewer loop, có self-test, và có đầy đủ artifacts để team tin tưởng khi chạy thật.

2 workflows đang được register
1 workflow exposed cho router
7 core nodes trong orchestrator
LLM optional với deterministic fallback

Control plane nhỏ và rõ

`main_graph` chỉ làm 5 việc: plan, route, dispatch, collect, finalize.

Domain logic nằm trong workflow

Mỗi workflow có metadata riêng, graph riêng, artifacts riêng và có thể lồng subgraph khác.

Review trước khi implement

`gameplay-engineer-workflow` không viết code ngay. Nó phải qua review loop tới khi score = 100.

Debug được, test được, mở rộng được

`graph_traversal.log`, `graph_state_debug.jsonl`, checkpoints và tests làm flow này vận hành được ở team scale.

Step 1 Prompt -> Tasks

Prompt lớn được chia thành từng task nhỏ, đủ hẹp để route đúng workflow.

Step 2 Tasks -> Specialist Workflow

`Workflow.md` quyết định khả năng discover, route và profile LLM của từng workflow.

Step 3 Workflow -> Reviewer / Self-test

Luồng nội bộ có thể tự review, sửa plan, regenerate code và test lại trước khi trả output.

Step 4 Artifacts -> Trust

Mỗi run sinh ra logs, summaries, checkpoints, code/tests, commit message và draft PR.

Why This Shift

Từ một agent làm tất cả sang nhiều workflow hợp tác

Điểm khác biệt không nằm ở chỗ "dùng LangGraph", mà nằm ở cách repo này tách phần orchestration khỏi phần domain execution. Kết quả là architecture dễ hiểu hơn, dễ test hơn và dễ gắn owner theo team hơn.

Từ behavior khó đoán -> execution có hợp đồng, có gate, có trace

Single all-in-one agent

Prompt-centric
  • Planning, routing, review và codegen trộn vào cùng một context lớn.
  • Khó xác định lỗi vì không biết sai ở bước plan, route hay implementation.
  • Khó test deterministic khi mọi thứ phụ thuộc vào một LLM flow duy nhất.
  • Khó chia owner vì không có boundary rõ giữa platform và domain expertise.
  • Khó quan sát nếu không có logs/artifacts tách theo từng stage.

Workflow-driven runtime

Graph-centric
  • Main graph chỉ orchestration: plan, route, dispatch, collect, finalize.
  • Mỗi workflow là một module runtime có metadata, state và graph riêng.
  • Subgraph ẩn như reviewer có thể chạy nội bộ mà không cần expose cho router.
  • Fallback vẫn chạy ngay cả khi planner/router/reviewer/codegen LLM unavailable.
  • Logs, checkpoints, artifacts giúp vận hành, debug và review output thực tế.
Repo Mapping

Repo này hiện thực hóa ý tưởng đó như thế nào?

Trong AgentSwarm, workflow không phải chỉ là prompt template. Nó là một contract có metadata, graph implementation, runtime wrapper, routing surface và output artifact rõ ràng.

Layer 01 Entrypoint

Nhận prompt, tạo `run_dir`, load workflows, build main graph, invoke runtime và ghi checkpoint summary.

Layer 02 Platform Core

`workflow_loader` parse `Workflow.md`, bind LLM profile, compile graph; `registry` quản lý lookup và fallback routing; `llm.py` trừu tượng hóa backend.

Layer 03 Main Orchestrator

`main_graph` giữ state chung, plan task, route workflow, dispatch subgraph và tổng hợp final response.

Control Plane

Main graph giữ orchestration, không ôm domain logic

`plan_tasks` Chia prompt thành 1 tới 5 implementation tasks. Nếu planner LLM không sẵn sàng, hệ thống fallback sang split deterministic.
`route_tasks` Ưu tiên router LLM, nhưng vẫn có fallback bằng keyword overlap trên `name`, `description`, `capabilities` của workflow metadata.
`select_next_task` + dispatch Chọn task có trạng thái `routed`, cắt state cần thiết cho task đó, rồi gọi workflow subgraph tương ứng.
`collect_task_result` + `finalize` Merge kết quả, cập nhật notes, lặp tới khi không còn task nào, sau đó ghi `summary.md` và trả final response.
Deep Dive

`gameplay-engineer-workflow` là nơi governance thật sự xảy ra

Workflow này cho thấy lợi thế lớn nhất của kiến trúc hiện tại: domain workflow có thể tự xây design context, tự review plan, tự sửa cho tới khi đạt chuẩn, rồi mới implement và self-test.

Reviewer là subgraph ẩn `gameplay-reviewer-workflow` có `exposed: false`, nghĩa là nó không được route trực tiếp từ user prompt, nhưng vẫn là một block chuyên trách được gọi bên trong engineer workflow.
Hai quality gates rất rõ Gate 1: plan phải được reviewer chấm 100/100. Gate 2: code phải compile và pass tests trước khi được phép prepare delivery.
Artifacts sinh ra rất thực dụng `design_doc.md`, `plan_doc.md`, `review_round_*.md`, `gameplay_change.py`, `test_gameplay_change.py`, `self_test.txt`, `commit_message.txt`, `pull_request.md`.

Trust Layer

Vì sao flow này đáng tin hơn khi đem ra team scale?

Repo hiện tại không chỉ “chạy được”. Nó đã encode sẵn nhiều cơ chế để giảm rủi ro khi dùng LLM trong runtime: deterministic fallback, thread-aware checkpoints, trace logs và per-task artifacts.

Deterministic fallback

`plan_tasks`, `route_tasks`, classification, review và code bundle generation đều có fallback. Nghĩa là flow vẫn chạy khi model unavailable.

Checkpoints theo thread

`build_runtime_config(thread_id)` và `InMemorySaver` cho phép lấy `get_state()` và `get_state_history()` sau mỗi run.

Graph-level observability

`graph_traversal.log` ghi ENTER / EXIT / ROUTE, còn `graph_state_debug.jsonl` lưu payload debug theo event id.

Tests đã validate behavior cốt lõi

`tests/test_runtime.py` cover load workflows, routing, nested subgraphs, end-to-end runtime và checkpoint state tracking.

Team Value

Ý nghĩa thực tế với team nếu đi theo hướng này

Dễ chia ownership

Platform team giữ orchestration và contracts. Domain team giữ từng workflow. Reviewer workflows có thể được team lead hoặc QA định nghĩa riêng.

Dễ mở rộng dần

Không cần làm “siêu agent” ngay từ đầu. Chỉ cần thêm workflow mới, metadata mới và route surface phù hợp.

Dễ review output

Artifacts theo task khiến việc review design, plan, test và PR draft trở thành quy trình quen thuộc với engineering team.

Dễ kiểm soát AI risk

LLM được dùng như capability trong từng step, không phải “black box” duy nhất điều khiển toàn bộ execution.

“Chúng ta không xây một agent thông minh duy nhất. Chúng ta đang xây một runtime để nhiều workflow chuyên trách cộng tác an toàn.”

Next steps nên ưu tiên

  • Thêm workflow mới theo domain như backend, frontend, docs, QA reviewer.
  • Giữ `Workflow.md` thật gọn để route surface dễ hiểu và dễ audit.
  • Xuất graph viewer trong CI để team luôn thấy topology mới nhất sau mỗi thay đổi.
  • Chuẩn hóa artifact contract để mọi workflow trả ra summary, validation và delivery assets nhất quán.
  • Chỉ expose workflow an toàn; internal reviewer / validator có thể giữ `exposed: false`.
